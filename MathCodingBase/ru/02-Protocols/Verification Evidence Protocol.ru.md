# Протокол свидетельств верификации

Верификация — это не предложение. Это артефакт.

Канонический путь артефакта внутри пакета — `verification.json`.

## Обязательные поля свидетельств

- имя модуля
- результат L1-врат
- команда SANY и код возврата
- команда TLC и результат
- проверенные инварианты
- проверенные временные свойства
- вердикт
- при вердикте `UNVERIFIABLE:*` — объект `human_review` (см. ниже)

## Вердикты

Поле `verdict` — закрытое перечисление:

- `VERIFIED`
- `NEEDS_REVISION`
- `UNVERIFIABLE:TOOL_MISSING`
- `UNVERIFIABLE:OUT_OF_SCOPE`
- `UNVERIFIABLE:DEFERRED`

`VERIFIED` требует реального запуска инструментов с записанными кодами возврата. `NEEDS_REVISION` требует записанного контрпримера.

## Подтипы UNVERIFIABLE

`UNVERIFIABLE` — размеченное объединение. Каждый пакет с пометкой «не верифицируется» обязан объявить, какой подтип применяется и почему.

### `UNVERIFIABLE:TOOL_MISSING`

Механический инструмент для этого вида артефакта существует, но сломан, отсутствует или недоступен в CI. Владелец пакета обязан починить или закрепить инструмент. Пример: jar-файл TLC удалили из образа CI; SANY сообщает о проблеме с путём к Java.

### `UNVERIFIABLE:OUT_OF_SCOPE`

Вид артефакта лежит за пределами механической верифицируемости. Примеры: качество прозы, эстетика интерфейса, этические решения, стратегическая расстановка приоритетов, дизайнерский вкус. Пакет обязан объявить именованного человека-рецензента с указанной областью компетенции и процессом ревью.

### `UNVERIFIABLE:DEFERRED`

Верификация требует данных, которых ещё нет: метрик продакшна, пользовательских исследований, нагрузочных тестов в масштабе, внешней обратной связи. Пакет обязан объявить владельца, срок и триггер пересмотра.

### `REJECTED` запрещён

Подтипа `UNVERIFIABLE:REJECTED` не существует. Агент, который решил, что верификация не нужна, обязан переформулировать задачу в меньшую проверяемую задачу. Пропуск верификации по личному суждению — методологический провал, а не «неверифицируемость».

## Обязательство компенсации

Каждый пакет `UNVERIFIABLE:*` обязан содержать в `verification.json` объект `human_review`:

```json
"human_review": {
  "by": "<имя или роль>",
  "process": "<процедура ревью, шаги, какие свидетельства должен произвести рецензент>",
  "trigger": "<когда проходит ревью: перед слиянием, перед релизом, по расписанию, ...>",
  "re_verification": "<необязательно: как и когда механическая верификация возобновляется>"
}
```

Отсутствующие или пустые поля проваливают `bin/validate-packet` для неверифицируемых пакетов. Это и есть цена объявления «не верифицируется»: вы должны назвать рецензента и процесс.

## Кто пишет файл

`bin/verify` — единственный допущенный писатель `verification.json` для `VERIFIED` и `NEEDS_REVISION`. Для `UNVERIFIABLE:*` агент сам пишет файл, когда ни один инструмент не запускается. Эта оговорка явная: неверифицируемость по определению означает, что инструмент не породил свидетельств, поэтому агент сам форматирует объяснение и блок `human_review`.

## Обёртки

Пользуйтесь обёртками репозитория, когда возможно:

```bash
./bin/tla-sany Model.tla
./bin/tla-tlc -config Model.cfg Model.tla
```

Обеим нужна переменная `TLA2TOOLS_JAR`.

Обёртка парсера запускает `tla2sany.drivers.SANY` из `tla2tools.jar`.

Если переменная не задана, обёртки автоматически ищут в обычных местах внутри репозитория и в пользовательских каталогах.
