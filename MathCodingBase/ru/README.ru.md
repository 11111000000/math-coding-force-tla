# MathCoding

`MathCoding` — методология плюс переносимый скаффолд. Она заменяет вайб-кодинг мат-кодингом и работает с любым агентом, способным прочесть файл и выполнить команду в оболочке.

Единица работы — **пакет задачи**. Это каталог, в котором лежат формулировка задачи, допущения, формальная модель, свидетельства верификации, план уточнения, карта трассируемости и реализация. Пакет — источник истины. Чат — нет.

## Зачем это нужно

Вайб-кодинг выдаёт правдоподобный код на основе текстового намерения. Код выглядит нормально, пока пользователь не кликнул дважды подряд, пока запрос не пришёл не в том порядке, пока деплой не разъехался на два узла. Код работал; модели кода не существовало.

Мат-кодинг переворачивает порядок. Сначала пишем модель. Проверяем её инструментом. Только потом пишем код, который её реализует. Из этого следуют четыре правила:

- допущения явные и с тегом (`user-confirmed`, `agent-inferred` или `open`)
- модель проверяется механически до того, как появится хоть строчка кода
- вердикт по модели — это `VERIFIED`, `NEEDS_REVISION` или `UNVERIFIABLE`, а не «выглядит нормально»
- методология описывает сама себя теми же артефактами, которые требует от вас

## Что лежит в репозитории

- каноническая структура пакета под `artifacts/<task-id>/`
- восемь обязательных артефактов пакета: `packet.json`, `problem.md`, `assumptions.yaml`, `Model.tla`, `Model.cfg`, `verification.json`, `refinement.md`, `traceability.json`, `implementation/`
- машино-читаемые схемы под `schemas/` для манифеста, допущений, отчёта о верификации и трассируемости
- три уровня механической верификации: SANY разбирает, TLC проверяет модель, TLAPS доказывает
- операционные сценарии под `bin/`: `mathpacket`, `verify`, `validate-packet`, `refine-from-model`, `tla-sany`, `tla-tlc`, `tla-tlaps`, `locate-tla2tools`
- файл `flake.nix`, который даёт воспроизводимую оболочку разработки с `jdk21` и публикует каждый сценарий как приложение
- адаптеры для opencode (`.opencode/`), Cursor (`adapters/cursor/`), Claude Code (`adapters/claude-code/`) и общий контракт (`adapters/generic/`)
- база знаний в стиле Obsidian под `MathCodingBase/` с самоприменением методологии

## Оперативный старт

```bash
# 1. Создать пакет
./bin/mathpacket demo-task "Demo Task"

# 2. Отредактировать problem.md, assumptions.yaml, Model.tla, Model.cfg
# 3. Сгенерировать скелет уточнения из разобранной модели
./bin/refine-from-model artifacts/demo-task

# 4. Прогнать все три уровня верификации
./bin/verify artifacts/demo-task

# 5. Проверить структурную корректность
./bin/validate-packet artifacts/demo-task

# 6. Писать код, потом снова верифицировать и валидировать
```

## Переносимость между агентами

Четыре каталога адаптеров, один контракт:

- **opencode** — `.opencode/` со скиллами, агентами и slash-командами
- **Cursor** — `adapters/cursor/.cursorrules` плюс руководство по работе
- **Claude Code** — `adapters/claude-code/CLAUDE.md` плюс slash-команды
- **универсальный** — `adapters/generic/README.md` описывает минимум, которому должен удовлетворять любой агент

Чтобы перенести `MathCoding` на новую среду, сохраните схемы, документы протоколов, конвенцию каталогов пакета и контракт механического верификатора. Замените промпты, slash-команды, права доступа к инструментам и обёртки. Всё, что лежит выше линии адаптера, остаётся неизменным.

## Конвейер верификации

Три уровня, запускаются через `./bin/verify`:

1. **SANY** разбирает `Model.tla`. Нет разбора — нет дальнейшей верификации.
2. **TLC** перебирает достижимые состояния в пределах границ из `Model.cfg` и проверяет каждый инвариант и каждое временное свойство.
3. **TLAPS** доказывает блоки `THEOREM` с помощью серверных решателей (Zenon, Isabelle, Z3, SMT-решатели).

Вердикт — одно из пяти:

- `VERIFIED` — каждый требуемый уровень прошёл по реальному выводу инструмента
- `NEEDS_REVISION` — как минимум один уровень не прошёл; в `verification.json` лежит контрпример
- `UNVERIFIABLE:TOOL_MISSING` — инструмент существует, но сломан или отсутствует в CI; доверие приостановлено до починки
- `UNVERIFIABLE:OUT_OF_SCOPE` — вид артефакта не допускает механической проверки (проза, интерфейс, стратегия, этика); доверие переходит к именованному рецензенту
- `UNVERIFIABLE:DEFERRED` — инструмент есть, но данных пока нет; доверие отложено до получения данных

Каждый пакет `UNVERIFIABLE:*` обязан содержать блок `human_review`: именованный рецензент, процедура, триггер. «Поверьте, всё нормально» — не вердикт.

## Текущее состояние

- Четыре пакета проходят механическую верификацию от начала до конца.
- TLAPS доказывает две настоящие `THEOREM` в `examples/minimal-spec`.
- `bin/refine-from-model` пересобирает скелеты уточнения и трассируемости из разобранной TLA+ модели.
- `bin/validate-packet` сообщает о структурных проблемах и о любых связях `source -> TODO`, оставшихся в трассируемости.
- `artifacts/ui-modal-react` поставляет редьюсер, представление и 16 проходящих тестов.
- Три адаптера работают: opencode, Cursor, Claude Code, плюс универсальный контракт.
