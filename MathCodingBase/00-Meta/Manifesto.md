# Manifesto

MathCoding replaces vibe-coding with math-coding.

## Thesis

Vibe-coding optimizes for local plausibility. You write the function, the test passes, you ship. The code looked right before you knew what "right" meant in this context — under concurrency, under partial failure, at three in the morning when the user's session has been idle for nine hours.

## Antithesis

Formal methods alone don't fix that. A specification written as prose, generated by the agent on the way to the code, and discarded once the code compiles, is ceremony. The mathematics runs alongside the design but never constrains it. You get the cost of rigor with none of the benefit.

## Synthesis

Math-coding makes mathematical artifacts the development substrate. The chat message is not the spec. The code is not the first artifact. The verified model is the gate, not a luxury. Four commitments fall out:

1. Every task becomes a task packet.
2. Every ambiguity becomes an explicit assumption.
3. Every non-trivial implementation sits behind a refinement plan.
4. Every claim about correctness carries evidence — a real exit code, a real counterexample, or a refusal to commit.

The fifth commitment is the one most people miss: the methodology has to apply to itself. If we can't describe the methodology through its own pipeline, we don't trust it as a methodology.
