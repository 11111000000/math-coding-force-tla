The Pipeline Конвейер 流水线

Seven stages from problem to verified code Семь стадий от задачи до проверенного кода 从问题到验证代码的七个阶段

Every task follows the same cycle. No step is optional. No artifact is transient. Каждая задача проходит один и тот же цикл. Ни один шаг не факультативен. Ни один артефакт не эфемерен. 每个任务都遵循相同的周期。没有可选步骤。没有临时产物。

01
Problem
Задача
问题
problem.md
Capture the task in plain prose. State desired outcome, non-goals, and what counts as success. Without this, the rest of the cycle has no anchor.
Опишите задачу в свободной форме. Укажите желаемый результат, что вне scope, и как понять, что задача решена. Без этого якоря остальной конвейер теряет опору.
用自然语言描述任务。明确期望结果、非目标与成功标准。否则后续阶段失去锚点。
02
Assumptions
Допущения
假设
assumptions.yaml
Every ambiguity becomes a tagged assumption: user-confirmed, agent-inferred, or open. Code never silently depends on an open one.
Каждая неоднозначность становится теговым допущением: подтверждённое пользователем, выведенное агентом или открытое. Код не должен молча опираться на открытое.
每个歧义转为带标签的假设:用户已确认、智能体推断或待定。代码不得隐式依赖待定假设。
03
Formalize
Формализация
形式化
Model.tla
Write the state machine in TLA+. State variables, transition relation, safety invariants, and liveness conditions are all explicit.
Опишите автомат на TLA+. Переменные состояния, отношения переходов, инварианты безопасности и условия liveness — всё становится явным.
用 TLA+ 写出状态机。状态变量、转移关系、安全不变量与活性条件皆为显式。
04
Verify
Верификация
验证
verification.json
SANY parses the spec. TLC exhaustively walks every reachable state and either accepts it or hands you a counterexample. No "expert gut".
SANY разбирает спецификацию. TLC проходит каждое достижимое состояние и либо принимает модель, либо показывает контрпример. Никакой «экспертной интуиции».
SANY 解析规约,TLC 穷举检查每个可达状态或给出反例。没有"专家直觉"。
05
Refine
Уточнение
精化
refinement.md
State variables map to program state. Actions map to procedures. Which invariants survive types, and which need runtime checks — all stated.
Переменные состояния отображаются в программу. Действия — в процедуры. Какие инварианты сохраняются типами, а какие требуют проверок в рантайме, всё прописано.
状态变量映射到程序状态。动作映射到过程。哪些不变量由类型保留、哪些需要运行期检查,一一说明。
06
Implement
Реализация
实现
implementation/
Code follows the refinement. Every action becomes a procedure; every invariant becomes a test or a runtime check.
Код следует плану уточнения. Каждое действие становится процедурой, каждый инвариант — тестом или проверкой в рантайме.
代码遵循精化方案。每个动作成为过程,每个不变量成为测试或运行期检查。
07
Trace
Трассировка
追溯
traceability.json
A machine-readable map from model element to code location. "Where in the code is Invariant N preserved?" gets a one-lookup answer.
Машиночитаемая карта: элемент модели — место в коде. Вопрос «где в коде сохраняется инвариант N?» имеет одно-точечный ответ.
模型元素到代码位置的机器可读映射。"代码中哪里保留了不变量 N?" 一查即得。

From TLA+ to TypeScript От TLA+ к TypeScript 从TLA+到TypeScript

The same state machine, formalized and implemented. Один и тот же автомат, формализованный и реализованный. 同一个状态机，经过形式化和实现。

Model.tla
VARIABLES state, pendingResult

Init ==
  /\ state = "closed"
  /\ pendingResult = "none"

Open ==
  /\ state = "closed"
  /\ state' = "opening"
  /\ UNCHANGED pendingResult

Confirm ==
  /\ state = "open"
  /\ state' = "confirming"
  /\ pendingResult' = "in-flight"
dialogReducer.ts
export function dialogReducer(
  state: DialogStateShape,
  action: DialogAction,
): DialogStateShape {
  switch (action.type) {
    case "OPEN":
      if (state.state !== "closed") return state;
      return {
        state: "opening",
        pendingResult: { kind: "none" }
      };
    // ... 10 actions total
  }
}
